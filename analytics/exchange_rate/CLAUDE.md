# analytics/exchange_rate/ — Panorama Cambial

Self-contained HTML report on Brazilian FX fundamentals (`reports/fx_report.html`) — single file, opens in any browser, safe to email. Reads from `macro_brasil` and `macro_international` (see [`domain/db/CLAUDE.md`](../../domain/db/CLAUDE.md) for schema-naming conventions). This is the applied/analytics branch of the exchange-rate work — see root `CLAUDE.md`'s `repository/` section for how it relates to the literature-curation and consolidated-synthesis branches.

## Generating the report

```powershell
uv run python jobs/update_db.py             # refreshes macro_brasil (cmb_reservas_bc, cmb_cambio_contratado, cmb_ptax, cmb_balanco_pagmt, cmb_fluxo_cambial, cmb_termos_troca, cmb_comex_*, atv_pib_usd)
uv run python jobs/update_international.py  # refreshes macro_international (cmb_reer, cmb_cot_fx, diferenciais_juros)
uv run python -c "from analytics.exchange_rate.generate_report import run; run()"
# → reports/fx_report.html (~50 KB)
```

Updating the DB first is optional — only needed for fresher data. `generate_report.py` alone re-renders against whatever is already in MySQL.

## Report architecture

Fixed template (`report.html`) with a `/*REPORT_DATA*/` marker inside a `<script>` block. `generate_report.py` loads each table, serializes to JSON, and does a plain `str.replace()` — no Jinja2, no templating engine. Every `_load_*()` function is independently try/excepted, so one missing or broken table degrades just that section (prints a warning) instead of failing the whole report.

Six tabs, real switching via JS `display` toggling (not scroll anchors): Balanço de Pagamentos (`tab-bop`, plus a companion BOP heatmap at `tab-heatmap`), Posicionamento do BCB (`tab-bcb`), Fluxo Cambial (`tab-flow`), Cotação (`tab-quotation`), Valuation (`tab-valuation`).

## Current data-quality gotchas

- **`cmb_ptax.ptax_venda` starts 1994-07-01, not 1984** — SGS 1 technically goes back to 1984-11-28, but everything before 1994-07-01 is denominated in extinct pre-Real currencies (Cruzeiro Real and, further back, Cruzeiro/Cruzado/Cruzado Novo), confirmed directly in the data by the 1994-06-30 (2750.00) → 1994-07-01 (1.00) break at exactly the Real's fixed launch parity. Pre-Real rows were deleted from the DB 2026-07-22 (not economically comparable without a conversion factor this project doesn't carry); `cmb_ptax.py`'s `_START_YEAR`/chunking was updated so a future `run(start="all")` backfill won't reintroduce them.
- **Financial Account sign flip is deliberate** (`_load_bop()` only — never touches the DB): `idp_exterior`, `portfolio_ativos`, `outros_inv_ativos`, `acoes_ativos`, `fundos_ativos`, `titulos_ativos_cp`, `titulos_ativos_lp`, `derivativos`, `ativos_reserva`, and `conta_financeira` are negated so the whole report reads "negative = USD outflow, positive = inflow" consistently, matching how Current Account already reads. Liabilities-side series (`investimento_direto_liquido`, `portfolio_passivos`, etc.) already publish in that convention and are left untouched. If you touch this function, re-derive the sign from a real month rather than assuming Assets/Liabilities are symmetric.
- **Units are not uniform:** `cmb_balanco_pagmt`, `cmb_reservas_bc`, `cmb_fluxo_cambial` store USD MM (divided by 1000 for display). `cmb_comex_pais`/`cmb_comex_fator_agregado`/`cmb_comex_produto` store raw USD (divided by 1e9).
- **Comex Stat breakdowns ≠ BPM6:** the by-country/by-factor/by-product tables use general-trade methodology (SISCOMEX), not BPM6 — their totals will not reconcile line-for-line with `cmb_balanco_pagmt.mercadorias_gerais`. Treat them as complementary cuts, not a decomposition of it.
- **Some "missing" months in Comex/product series are real zeros, not gaps** (e.g. `demais_import` in the Fator Agregado breakdown, `minerio_ferro_import`/`petroleo_export` in the product breakdown — confirmed as months with zero transactions, not pipeline failures). `generate_report.py` `fillna(0)`s these deliberately; do the same for any new derived series built on top.
- **`lucros_reinvestidos` (BCB SGS 22815) has no data 1999–2010** (confirmed 404 from the BCB API, not a pipeline bug) — already `fillna(0)`'d before summing into `lucros_dividendos`.
- **No "gross reserves" series** — SGS 13127 (`reservas_brutas_usd`) times out consistently (wrong/discontinued code); resolved by using the liquidity concept (`cmb_reservas_bc.reserves_liquidity_daily`) plus its detailed components instead. Not a gap to revisit.

## `models/` — research track (not wired into any report)

Statistical models testing FX theory directly against this database — distinct from `generate_report.py`, which only displays raw series, no estimation. Goal (stated 2026-07-22): understand where simple models fail at explaining USD/BRL. Originally scoped around three candidate channels — carry, terms-of-trade, fiscal credibility — but terms-of-trade and fiscal credibility were dropped 2026-07-23 (see below) in favor of going straight to PPP as the state-space equilibrium anchor. See `models/DATA_REQUIREMENTS.md` for the (now partly superseded) data-availability mapping across all three.

- **`uip_model.py`** (`uip_model.md`) — the strict UIP test (β=1) on the short-rate nominal differential alone. OLS + GARCH(1,1) + 2-regime Markov-switching. Result: consistently negative β (wrong sign vs. UIP), matching the Fama-puzzle/Araújo et al. literature — though the regime assignment doesn't replicate their exact low-vol/high-vol split.
- **`carry_model.py`** (built 2026-07-22 — **`carry_model.md` not written yet**; do that, following `uip_model.md`'s pattern, before calling this model "done"). Broader, less restrictive than `uip_model.py`: 4 specs crossing horizon (short-rate Selic−Fed Funds vs. BR−US 2y point) × construction (nominal vs. real ex-post), plus a "dynamic" spec testing the *change* in the short-rate differential (`diff_a_nom_chg_lag1`) alongside a new `gl_mc` "global monetary cycle" variable (US 2y − Fed Funds, the US front-end curve slope). Sample windows differ by spec and were set from verified data, not assumption: short-rate specs run 1999-03→today; the BR 2y leg (`base_mercado.interest_rates`, curve `PREJS`, tenor `24M`) only starts 2006-01 — confirmed against Tesouro Direto's actual historical bond file that no genuine 2-year point existed before then (checked live, not assumed). All 5 specs: weak/insignificant, R² ≈ 0.002–0.004 — carry is not, on its own, a workable predictor in this data, consistent with `uip_model.py`'s own finding.
- **`ppp_equilibrium.py`** (built 2026-07-23) — not a regression, a data/dashboard generator: builds the relative-PPP equilibrium candidate (headline IPCA index ÷ headline CPI index, anchored to actual PTAX at a selectable base month) directly named in the state-space concept note's open thread below, and renders it to `referencia/ppp_dashboard.html`. Sample 1994-07→today (383 months), the full range where PTAX/IPCA/CPI all overlap. Rerun via `uv run python -c "from analytics.exchange_rate.models.ppp_equilibrium import run; run()"` — refetches BR IPCA + PTAX from MySQL and US CPI live from FRED (not cached in DB) each time. Same day, also extended to fetch four candidate channels for explaining the PPP deviation (now regressed — see `bayesian_deviation_model.py` below): `carry` = `diferenciais_juros.diferencial_nominal` (Selic − Fed Funds, from 1999-03 — Selic-target regime start, not a fetch gap), `tot` = `cmb_termos_troca.termos_de_troca_funcex` (Funcex PX/PM via IPEADATA, real history from 1978 trimmed to 1994-07+), `breakeven` = 10y bond-implied inflation expectation, `PREJS − NTNBJS` @ 120M tenor from `base_mercado.interest_rates` (**external** fund-ops schema, not this project's ETL — read live via `MySQLDataRequester`, same pattern `carry_model.py` already uses for its BR-2y carry spec), from 2006-01, and `fiscal` = `cmb_risco_pais.cds_5y_usd` (Brazil 5Y CDS USD, manually ingested from investing.com exports, from 2007-12). All four are left-joined onto the core monthly index (null before each one's own start — no back-fill), and charted as their own dashboard panels. `breakeven` added 2026-07-23 specifically as a long-term-inflation-expectation proxy, after confirming BCB Focus's own longest horizon (`expc_focus`, IPCA 24m) only reaches back to 2021-03 — too short to be useful. Also carries forward a real, documented fix for the confirmed `PREJS@120M` data bug (two bad windows, `2010-01-22`–`2010-02-05` and `2010-03-02`–`2010-03-04`, values ~3% vs. a true ~13%, isolated to that one curve/tenor — verified 60M and `NTNBJS`@120M clean over the same period) via `_PREJS_120M_BUG_WINDOWS` + linear time-interpolation in `_load_breakeven()` — this is the "fix it in production" ask from `state_space_equilibrium_model.md`'s inline note, finally done as reusable code rather than the prior throwaway plotting script (though the fix still lives in this project's code, not in the external table itself). Also gained `compute_deviation(df, base_month)` (the same base-month-invariant deviation math the dashboard's JS computes client-side, now available server-side for `bayesian_deviation_model.py` to reuse) and a `target`/`breakeven_gap` column pair (`macro_brasil.inflc_meta`, BCB SGS 13521, the CMN inflation target — annual, forward-filled to monthly; `breakeven_gap = breakeven - target`, the "de-anchoring gap" used as an alternative regressor construction in the Bayesian model).
- **`bayesian_deviation_model.py`** (built and fit 2026-07-23) — Bayesian regression of the PPP deviation on carry/terms-of-trade/breakeven/fiscal, "attempt one" ahead of the Kalman-filter state-space model ("attempt two", concept-stage below). Full design + results in `models/bayesian_deviation_model.md`. Pre-registered an ADF/KPSS stationarity check before writing any model code — result: `deviation`, `carry`, `tot` are I(1), `breakeven`/`fiscal` are I(0), so the model uses first differences uniformly (a deliberate "attempt one" simplification, not a claim that's optimal). Fit via PyMC/NUTS (4 chains, r-hat=1.00 throughout): **terms-of-trade is the one consistently-signed result across both the short primary sample (2008-02→today, P(β<0)=0.82) and a long carry+tot-only robustness sample (1999-05→today, P(β<0)=0.83)**; fiscal risk (CDS) has the strongest single-spec sign probability (0.94) but no longer-sample check exists for it; carry shows no reliable signal and its sign isn't even stable across samples, extending `uip_model.py`/`carry_model.py`'s own finding; **breakeven (and the breakeven-minus-target de-anchoring gap, which turned out empirically ≈identical once differenced) came back with the opposite sign from what was expected** — a genuinely open, unresolved finding (candidate explanations: a hawkish-response/credibility-premium channel, or simultaneity the 1-month lag doesn't fully clear), not something to wave away. Normal vs. Student-t error comparison favored Normal (no evidence of fat tails at this sample size). Added `pymc`+`arviz` as project dependencies (`uv add pymc arviz`) and, mid-session, a system-level MSYS2/MinGW-w64 g++ compiler (see `AMBIENTE.md`) after the no-compiler PyTensor fallback proved too slow — cut each model fit from 10+ minutes to ~35-40 seconds. `run()` saves each fit's posterior trace to `bayesian_results/*.nc` (gitignored — regeneratable, ~30-40 MB apiece) plus a summary CSV; `run()` must be called at least once before anything below can read them back. **`build_dashboard_payload()`** (added same day) loads those saved traces (`load_saved()`, `az.from_netcdf` — no refit) and derives everything the dashboard's "Bayesian Model" tab needs: per-spec diagnostics tables, a historical-fit reconstruction (posterior mean + 94% credible band, in both Δdeviation and cumulative-level units, anchored to the actual deviation level one month before the primary spec's sample starts), a point-estimate decomposition (each channel's cumulative contribution via its posterior-mean β, plus a residual term — sums exactly back to the actual cumulative change, verified to ~1e-4), posterior histograms, and a cross-spec forest-plot structure. **`render_dashboard()`** is now the canonical way to generate the full dashboard (both tabs) — it calls `ppp_equilibrium.load_data()`/`build_payload()` for tab 1 and `build_dashboard_payload()` for tab 2, then a single `ppp_equilibrium.render()` call fills both template markers at once.

**Follow-up, same day: is the drift (α) real, or an artifact?** The primary spec's α (0.219/month) compounds into most of the current PPP gap — in the level bridge, "baseline" is +R$1.54 of the ~R$1.58 gap between equilibrium (R$3.48) and actual PTAX (R$5.06), dwarfing all four named channels (currently netting to ≈R$0.00). User pushback: "the PPP should be the only directional systematic drift along the time." Two tests, full detail + numbers in `models/bayesian_deviation_model.md`'s "Follow-up: testing the drift directly" section:
  - **No-intercept spec** (`primary_no_alpha`, α forced to 0): the 4 betas came back statistically unchanged — the drift didn't disappear, it just relocated from "baseline" to "residual." Why: standardized (z-scored) regressors have sample mean exactly 0 by construction, so none of them can ever explain delta_dev's actual non-zero mean (+0.222/month) — that mean can only land on the intercept, or, with none available, the residual.
  - **Error-correction spec** (`primary_ecm`, "Option B", user's own suggestion): adds `deviation_lag1` (the lagged deviation *level*) alongside the 4 channel deltas, α kept — but critically must be passed via the new `raw_cols=["deviation_lag1"]` argument to `fit_regression()` so it stays in native (non-zero-mean) units; a first attempt wrongly standardized it too, repeating the exact same mistake, caught and fixed before reporting results. Result: ρ (β_deviation_lag1) = −0.014, P(ρ<0)=89.2% (suggestive, not "reliable" by this model's own 94%-HDI bar) — implying a median half-life (among negative draws) of **~44 months**, matching Rogoff's classic "PPP puzzle" range (3-5 years) almost exactly. α *increased* (0.219→0.424) rather than shrinking — not a bug: α now means "drift at deviation=0," and since the sample's average deviation was well above zero with ρ weakly negative, some of the old "average drift" was actually reversion-from-an-elevated-start (a drag), so removing it pushes the zero-point intercept up.
  - **Net honest read**: weak-but-real evidence of PPP-consistent mean-reversion, at a textbook-matching magnitude — but even accounting for it, a sizeable unexplained constant pull toward BRL weakness survives. Neither test is wired into the dashboard or the Excel workbook yet (discussion-only round); the ~44-month half-life is flagged as a candidate concrete input for calibrating the state-space model's equilibrium-pull speed, next session.

Don't restart them without re-confirming scope. (Note: this is distinct from the `tot`/`fiscal` *data series* now fetched in `ppp_equilibrium.py` above — those are raw inputs for the planned Bayesian regression, not the abandoned standalone models, which would have been full estimation exercises in their own right.)
- **`export_excel_audit.py`** (built 2026-07-23) — generates `reports/ppp_model_audit.xlsx` (gitignored), a live-formula audit workbook so the whole pipeline can be checked by hand outside Python: `Raw Data` (original series, static) → `PPP Equilibrium` (equilibrium/deviation formulas, base-month dropdown) → `Model Parameters` (Bayesian posterior means from `bayesian_results/primary_breakeven_summary.csv`, hardcoded — **explicitly not re-estimated in Excel**, per user call: "it's not necessary to construct the OLS, just plug the parameters"; standardization mean/stdev *are* real `AVERAGE`/`STDEV.S` formulas, since that's mechanical data prep, not model-fitting) → `Deltas` (differencing, 1-month lag, z-scores) → `Decomposition & Bridge` (contribution/cumulative math plus the lvl0..lvl6 sequential level-bridge from `bayesian_deviation_model.py`, reconstructing the actual PTAX rate exactly — verified independently in a pandas shadow calc, max abs error ~1e-15). All sheets share one date-row backbone (row 2 = 1994-07, row 384 = 2026-06) so the same row number means the same month everywhere. Rerun via `uv run python -c "from analytics.exchange_rate.models.export_excel_audit import run; run()"` — requires `bayesian_deviation_model.run()` to have been run at least once first (reads the saved summary CSV, doesn't refit). **Two bugs fixed same day, both caught by direct user review of the generated file**: (1) the IPCA/CPI index headers said "(base=100)", which is misleading since the sample's first visible row (1994-07) already reads ~916 — the index is genuinely seeded at 100 in 1994-01 (before PTAX data starts and gets joined in), and six months of pre-Real-Plan hyperinflation compound before the row becomes visible; header reworded, README section added explaining this is real history, not a data error. ### State-space equilibrium model (concept stage — no code yet)

`referencia/state_space_equilibrium_model.md` proposes treating USD/BRL "fair value" as a Kalman-filter unobserved state, with carry/terms-of-trade/fiscal-credibility as pull variables (BEER/NATREX-style). The doc's original build sequence wanted `terms_of_trade_model.py` and `fiscal_credibility_model.py` built as standalone models first — **both abandoned 2026-07-23** (explicit user call, time-constrained) in favor of starting the measurement equation directly from PPP.

An interactive simulator for the doc's 3 toy equations (measurement/transition/deviation) was built 2026-07-22 and lives at `referencia/state_space_simulator.html` (self-contained, open directly in a browser) — also published as a Claude Artifact at `https://claude.ai/code/artifact/40710b8c-8ee7-4769-9c35-02a7e9180d16` for sharing, but the repo copy is now the source of truth for edits. `referencia/ppp_dashboard.html` (see above) is the first real-data companion to it.

Open threads on this concept note (next session should resolve or explicitly re-scope, not silently drop):
- **Partly resolved 2026-07-23**: the user settled the measurement-equation equilibrium question in favor of PPP — specifically the BRL/USD price-level ratio (headline IPCA/CPI), reasoning (i) the goal is understanding the *nominal* rate, not the real one, and (ii) the nominal rate empirically oscillates around this ratio over time. `ppp_equilibrium.py`/`ppp_dashboard.html` implement this as **relative** PPP (ratio of price-level *changes* since a selectable base month, anchored to the actual PTAX rate at that month) rather than absolute PPP (which would need internationally comparable price *levels*, not just indices) — the base-month dependence this introduces is real and is surfaced directly in the dashboard, not hidden. Still open: whether/how to fold this into the Kalman-filter measurement equation itself (the dashboard is a standalone exploration, not yet wired into any estimation code), and the second inline question — whether MA/time-varying parameters should be added, to test the hypothesis that the "important" fundamental changes over time — the same concern the doc's own Scapegoat Theory caveat already names.
- **Sign-convention consistency, unresolved**: the doc's `rer` follows BIS REER convention (rise = appreciation), but the simulator's deviation chart and β₁/β₂ defaults were built assuming the opposite (rise = more depreciated). The user's only comment so far — "it's ok for a fall to represent a depreciation" — accepted that framing in isolation but didn't confirm whether the β sign defaults should still be flipped for internal consistency with the REER convention.
- **Permanent vs. temporary pull-variable classification** (which of carry/terms-of-trade/fiscal-risk belongs in the transition vs. deviation equation) was discussed conceptually only — order of integration (I(1) vs I(0)), ADF/KPSS unit-root tests, Beveridge-Nelson trend/cycle decomposition — none of those tests have actually been run on the real series yet.
- **New concrete input, 2026-07-23**: the Bayesian model's error-correction follow-up (`primary_ecm`, see `bayesian_deviation_model.py` above) found weak evidence that the PPP deviation mean-reverts with a median half-life of **~44 months**, matching the Rogoff PPP-puzzle literature. That's a real, data-derived candidate for calibrating (or as a prior on) how fast the state-space model's equilibrium-pull mechanism should be — bring it into the design discussion rather than re-deriving from scratch.
